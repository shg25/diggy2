# レッスン07: 作ったものの寿命に責任を持つ — 増え続けるスプライト

対応コミット: `6c4a78b`

## 何が問題だったか

レッスン06の余談で見つけたリークを直す回。まず**測った**。
ゲームを3回続けてプレイして、自機ショットのプール配列と
画面のDOM要素数を数えると:

| プレイ回数 | ショット配列(種類ごと) | 画面のDOM要素 |
|-----------|----------------------|--------------|
| 1回目終了後 | 3 / 3 / 3 | 14 |
| 2回目終了後 | 6 / 6 / 6 | 24 |
| 3回目終了後 | 9 / 9 / 9 | 34 |

1プレイごとに、使われないスプライトが9個ずつ積もっていく。
発射処理は配列の先頭3個しか見ないので、ゲームは正常に動き続ける。
**動き続けるからこそ、誰も気づかない。** 熱中して30回遊んだ友人の
ブラウザでは、270個の見えないDOM要素が漂っていたことになる。

## 原因と直し方

原因は `goReady()`(ゲーム開始処理)が毎回 `defineJikiSh()` を呼び、
そのたびにプール配列へ新しいスプライトを push していたこと。

面白いのは、**設計自体は正しかった**ことだ。ショットは
「3発ずつ作り置きして、hide/show で使い回す」——これは
オブジェクトプールという立派なパターンで、毎フレーム生成・破棄
するより効率がいい。当時の自分は(たぶん無自覚に)良い設計を
していた。壊れていたのは一点、**プールを毎ゲーム作り直していた**こと。

だから修正も一点。「初回だけ作る」ガードを入れる。

```js
function defineJikiSh() {
	if (jikiSh1.length > 0) return; // プールは初回だけ作る
	...
}
```

## 計測が2匹目を連れてきた

修正して再計測したら、ショット配列は 3/3/3 で一定になったのに、
DOM要素が 14→15→16 と**まだ1個ずつ増えていた**。

犯人は「READY?」「GO!!」などを出すテキスト表示。`goReady()` が
毎ゲーム `newTxtAlert()` を呼び、古いテキストを隠したまま新しい
`DGE.Text` を作っていた。ショットプールと全く同じ症状。
こちらも「2回目以降は初回のものを使い回す」ガードで修正した。

**修正の正しさは目視より数字で確認できる。** 最終計測では
何回遊んでもショット配列 3/3/3、DOM要素も一定になった。

## 教え

1. **new したものには寿命の計画を。** 作る場所を書いたら
   「これはいつ消える? 誰が消す?」を自問する。答えが
   「わからない」なら、それはリーク予備軍。このゲームの
   スプライトたちの寿命は3パターンに整理できる:
   - **一生もの**(背景、ショットプール) — 一度だけ作る
   - **ゲーム1回分**(自機、ボス、とぅと郎) — 開始で作り、勝敗で消す
   - **秒単位**(雑魚、弾、アイテム) — 湧いて、画面外か撃破で消える

2. **リークは「動かなくなるバグ」ではなく「遅くなるバグ」。**
   エラーも出ず、短時間のテストでは見えない。だから
   「N回繰り返して数を数える」計測が唯一の検出法。
   直す前に測り、直した後にまた測る。数字の差分が修正の証明になる。

3. **悪いのはパターンではなく使い方。** 「コピペが悪い」
   「グローバルが悪い」と同じ調子で「作り置きが悪い」のではない。
   プール自体は優れたパターン。位置づけを理解して使えば武器になる。

## 余談

当時の自分へ: このプール設計、本当に悪くない。3発制限があるから
連射が「トトトッ、(装填)、トトトッ」というリズムになるのは、
実はプールの空きを待っているからで、それがゲームの手触りに
なっている。バグとゲーム性は紙一重。
