## フォーマッタとは

コードの**見た目**(インデント・スペース・改行・引用符)を、
設定に従って機械的に統一するツール。lint(レッスン12)が
「バグの芽」を見るのに対し、フォーマッタは「見た目」だけを見る。
実行しても動作は変わらない——構文木が同じになることを
ツール自身が保証している。

## なぜ入れるのか

このコードには `width : 8` という独特のスペースがあった。
当時の自分の癖だ。悪くはない。だが「癖」は、共同作業の瞬間に
コストになる。友人がPRをくれたとき、diffの半分が
「スペースの流儀の違い」だったら、本質が埋もれる。

フォーマッタの本当の価値は「綺麗になる」ことではなく、
**見た目の議論が構造的に不可能になる**こと。
「どっちのスタイルが正しいか」という、勝者のいない議論に
費やす時間が全員からゼロになる。設定ファイル(.prettierrc)に
決定を書いたら、以後は機械が執行する。

## 導入の意思決定

```json
{
	"useTabs": true,      // 元のコードがタブだったので維持
	"singleQuote": true,  // 元のコードがシングルクォートだったので維持
	"printWidth": 100     // 定義テーブル(レッスン04)が1行1種類を保てる幅
}
```

設定は最小限にして、残りはPrettierの既定に従う。
こだわりを設定に詰め込むほど「うちだけの方言」が復活し、
フォーマッタを入れた意味が薄れる。

.prettierignore には lib/(借り物)と *.md(レッスン文書の
手書きの呼吸を保つ)を入れた。**フォーマッタにも管轄外がある。**

## 一斉再整形という「儀式」

導入コミットは全ファイル一斉再整形(+631/-408行)になる。
これは避けられない通過儀礼で、やるなら:

1. **単独のコミットにする**(機能変更と絶対に混ぜない。
   レッスン02で学んだ「動かす変更と変える変更を混ぜない」の極端形)
2. **直後に全テストを回す**(理論上は安全でも、実証はする)
3. **門番に format:check を追加**して、以後の手書き逸脱を防ぐ

`key : value` の間隔は失われた。当時の手触りをひとつ
差し出した対価に、これから書くすべてのコードの一貫性を得た。
git履歴には当時の姿が残っている。

## 教え

1. **スタイルは「決める」ものであって「議論し続ける」ものではない。**
   どのスタイルでも慣れれば読める。読めるなら、決めて固定して、
   頭のリソースを中身に使う。

2. **一斉変更は単独コミットで。** 「ついでにバグも直した」を
   全ファイル再整形に混ぜたら、レビューも bisect も死ぬ。

3. **ツールの分担を知る。** format(見た目) → lint(バグの芽) →
   test(挙動) の3層は守備範囲が違い、全部必要。

## 余談

Prettierの思想は「設定項目を意図的に少なくする」ことで有名だ。
選択肢が少ないから議論も少ない。道具の設計自体が
「見た目の議論を終わらせる」という目的に奉仕している。
道具を選ぶときは機能一覧ではなく思想を見る。
