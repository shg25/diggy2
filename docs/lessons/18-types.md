# レッスン18: 型は仕様書 — JSDoc + TypeScript チェッカー

対応コミット: `8513e88`

## ビルドしないTypeScript

TypeScript と聞くと「.ts に書き換えてコンパイルする」を想像するが、
実はもうひとつの使い方がある。**JSファイルはそのまま**、型情報だけを
JSDoc コメントで書き、TypeScript コンパイラを検査専用
(`tsc --noEmit`)で走らせる方式だ。

```js
/** @param {number} keyCode */
export function eventKeyDown(keyCode) {
```

これを選んだ理由は、この旧作の性質にある。ビルド工程を入れると
「ファイルをそのままサーバーに置けば動く」という20年間変わらない
シンプルさが失われる。型の恩恵だけを受け取り、配信は素のまま。
将来の新機能版で本格的な .ts 化をやるとしても、この JSDoc は
そのまま型定義として通用する(TSはJSDocを完全に理解する)。

## 借り物との境界線: dge.d.ts

エンジン(lib/diggy/)は型を知らない2010年のコードだ。触らない
方針(レッスン02)のまま型検査と共存するために、**ゲームから見える
表面だけ**を型宣言ファイル types/dge.d.ts に書いた。

```ts
interface DGESprite {
	on(event: string, handler: (this: DGESprite) => void): DGESprite;
	...
}
```

見どころは `this: DGESprite`。エンジンはイベントハンドラを
スプライト自身を this にして呼ぶ——という**暗黙の約束**が、
初めて文章になった。おかげで10箇所ある `function () { this.set(...) }`
に1つも注釈を書かずに、this の型が通った。
**型宣言は、コードに書かれていなかった約束を書き出す作業**だ。

## strict で出た45件のエラーが教えてくれたこと

`strict: true` で初回実行すると45件。内訳が面白い。

- **引数の型が不明(28件)**: 「この関数、何を渡せばいいの?」が
  20年間、読む人の推測に任されていた箇所の一覧
- **変数の型が不明(16件)**: `export let jiki;` のような
  「後で入れる」変数たち
- **window.ridge が存在しない(1件)**: デバッグ窓口は自分で
  生やしたプロパティなので、Window への型拡張を宣言して解決

注釈を書く過程で、型がドキュメントになる瞬間が何度もあった。

```js
/** @type {'r' | 'l' | undefined} 回転方向 */
let bossTurn;
```

「r か l の文字列が入る」という頭の中にしかなかった仕様が、
リテラル型として明文化された。もし `bossTurn = 'right'` と
書き間違えたら、実行する前に赤線が引かれる。

## 妥協も記録する

- `get(key)` の戻り値は `any` にした。エンジンのプロパティバッグは
  動的で、正確に型付けるにはエンジン改修が要る。**全部を厳密に
  やろうとして挫折するより、境界に any を置いて内側を守る**
- `/** @type {DGESprite} */ export let jiki;` は実際には生成前は
  undefined なので、少し嘘をついている。正直に
  `DGESprite | undefined` と書くと全使用箇所に「undefined かも」
  チェックが要求され、挙動保存の範囲を超える。この嘘は
  ライフサイクル整理(いずれ)で返済する

## 教え

1. **型は未来の読者への仕様書。** コメントと違って、嘘をつくと
   機械が指摘してくれる。腐らないドキュメント。

2. **境界から型を付ける。** まず外部(エンジン)との境界に宣言を書き、
   次に自分のコードの公開面(export)、最後に内部。逆順だと
   エラーの洪水に溺れる。

3. **any は敗北ではなく防波堤。** 動的な領域を any で囲い、
   静的な領域を広げていく。ゼロか百かではない。

## 余談

型注釈を書き終えて眺めると、このゲームの関数はほぼ全部
`number` と `DGESprite` しか受け取らない。プリミティブな
シューティングゲームの素直さが、型のシンプルさに表れている。
型を書くと、設計の姿が鏡に映る。
