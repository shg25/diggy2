# レッスン08: 比較は厳密に — `==` がやっている余計な親切

対応コミット: `d1bb926`

## 何が問題だったか

コード中の比較はすべて `==` / `!=` (82箇所)だった。

JavaScriptの `==` は、型が違うと**勝手に型変換してから比べる**。

```js
0 == ''        // true (!?)
0 == '0'       // true
'' == '0'      // false (上2つと矛盾)
null == undefined  // true
1 == '1'       // true
```

この「親切」は、バグを隠す方向に働く。たとえば
`velJiki == 5` は、`velJiki` にうっかり文字列 `'5'` が入っていても
true になり、その場では動いてしまう。壊れるのはもっと先の、
原因から遠い場所——足し算したら `'55'` になった、とか。

`===` / `!==` は型変換をしない。型が違えば問答無用で false。
「数字のつもりだったのに文字列だった」がその場で発覚する。

## 直し方 — 機械的な置換ではない

一括置換したくなるが、**1箇所ずつ「両辺の型」を確認しながら**が正しい。
`==` の型変換に**依存している**コードがあった場合、`===` にすると
挙動が変わるからだ。今回の82箇所を精査した結果:

- 数値どうし: `stepFlg === STEP_BATTLE`, `keyCode === 32` など大半
- 文字列どうし: `bossTurn === "r"`, `turn === "on"` など
- undefined と文字列: `sprite.get('tag') !== 'boss'`
  (非ボスは tag 未設定 → undefined。`!=` でも `!==` でも
  「undefined は 'boss' ではない」で結果は同じ)

エンジンの `get`/`set` が値を素通しで保存すること
(`this.data[key]` に入れたまま返す)も確認した上で、
全箇所とも型変換に依存していない=置換して安全、と判断した。

依存箇所が見つかったら? そこは置換せず、なぜ型が揺れるのかを
直すのが先。幸い今回はゼロだった。

## 教え

1. **`===` をデフォルトに、`==` は使わない。**
   「`null` と `undefined` をまとめて弾きたいときだけ `== null`」
   という流派もあるが、初心者のうちは例外を作らないほうが安全。

2. **「一括置換で安全」は調査の結果であって、仮定ではない。**
   82箇所の精査があって初めて sed 一発が許される。
   調査なしの置換は、動いているコードへのロシアンルーレット。

3. **現代なら:** ESLint の `eqeqeq` ルールがこれを強制してくれる。
   もう人間が目を光らせる仕事ではない。

## 余談

精査中に見つけたもの: `makeBossSh1` に `stepFlg == 13` という
比較があった(現在は `=== 13`)。13 は `STEP_BATTLE` のこと。
すぐ上の行では `STEP_BATTLE` と書いているのに、ここだけ生の数字。
定数を定義した本人が定数を使い忘れる——マジックナンバーの
レッスンをやる日が来たら、こいつが主役になる。
