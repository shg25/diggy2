# レッスン02: プロジェクトの構え方 — 名前・借り物・README

対応コミット: `77cdab0`

## 何が問題だったか

当時の構成:

```
diggy/upload/
├── 1/            ← ゲーム本体。「1」って何?
│   ├── index.html
│   ├── js/
│   └── gfx/
├── src/          ← 借り物のDiggyエンジン
└── LICENSE       ← 何のライセンスか分からない位置
```

新しい構成:

```
diggy2/
├── index.html    ← ゲームの入口
├── js/           ← 自分たちのコード
├── gfx/          ← 画像アセット
├── lib/diggy/    ← 借り物のエンジン(LICENSE込み)
├── docs/         ← このレッスン集
└── README.md     ← プロジェクトの説明
```

## 教え

1. **ディレクトリ名は中身を語らせる。**
   「`1`」はおそらく「バージョン1」のつもり。だがバージョン管理は
   バージョン管理システム(当時ならSVN、今ならgit)の仕事で、
   ディレクトリ名でやると「`2`を作った瞬間に`1`との差分が
   追えなくなる」。名前は番号ではなく役割を書く。

2. **自分のコードと借り物は物理的に分ける。**
   `src/` は慣習的に「このプロジェクト自身のソース」を意味する。
   そこに他人のエンジンが入っていると、どこからが自分の責任範囲か
   分からなくなる。第三者のコードは `lib/` や `vendor/` に隔離して
   LICENSEも同梱する。これは法的な作法であると同時に、
   「借り物は原則いじらない。いじるなら意識的に」という
   自分への線引きでもある。

3. **READMEは未来の自分への手紙。**
   何のプロジェクトか、どう起動するか、どう操作するか。
   書くのに5分もかからないが、これがないと20年後の自分は
   ディレクトリ構成の考古学から始めることになる(実際なった)。

## 余談

移動はすべて `git mv` で行ったので、gitは全ファイルを
rename(100%一致)として追跡している。`git log --follow js/ridge.js`
とすれば `1/js/ridge.js` 時代の履歴も追える。
「動かすだけの変更」と「中身を変える変更」を同じコミットに
混ぜないこと。混ぜるとrename検出が壊れ、レビューも困難になる。
