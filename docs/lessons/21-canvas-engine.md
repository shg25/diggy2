# レッスン21: 器から作る — canvasミニエンジン

対応コミット: `4603c7e`

## 20年ぶりに「エンジンを書く側」へ

JIB-FREAK MOBILE の器(タイトル画面まで)を建てた。
Diggy は使わない。エンジンは自作、フレームワークもなし。
必要だったのは4部品、合計150行ほどだ。

```
jibfreak/js/engine/
├── loop.js     ゲームループ(requestAnimationFrame + 経過時間)
├── screen.js   画面(論理600x400のcanvas + 自動拡縮)
├── input.js    入力(キーボードとタッチを「意味」に束ねる)
└── assets.js   素材読み込み(全部揃ってから起動)
```

「エンジン」というと大袈裟に聞こえるが、正体はこれだけ。
20年前は Diggy という借り物のブラックボックスだったものの中身が、
自分で書くと150行で見通せる。**道具は、一度作ると怖くなくなる。**

## DOM から canvas へ — 20年分の設計の違い

| | Diggy (2010) | 自作 (2026) |
|---|---|---|
| 描画 | DOM要素を動かす | canvasに毎フレーム全部描き直す |
| ループ | setInterval 30fps | requestAnimationFrame + dt |
| 速度 | フレーム基準(1フレームで2px) | 時間基準(1秒で60px) |
| 画面 | 固定600x400 | 論理600x400を端末に合わせ拡縮 |

一番大事な違いは**時間基準**。Diggy の「1フレーム2px」は、
30fpsが保証されないと速度が変わる。dt(前フレームからの経過秒)に
速度を掛ける方式なら、60Hzでも120Hzでも重い端末でも、
1秒後には同じ場所にいる。MOBILEを名乗るなら必須の土台。

入力も設計から変えた。ゲーム側は `wasPressed('action')` と
「意味」を聞くだけで、それがスペースキーかEnterかタップかは
input.js しか知らない。キー配置変更もゲームパッド対応も、
このファイル1枚で済む(classicのキーコード直書きへの答え)。

## 移植性の証明: flow.js は1文字も変えていない

新版の flow.js は classic からの**そのままコピー**だ。
状態機械はDGEにもDOMにも依存しない(レッスン14)ので、
エンジンが DOM→canvas に総取り替えになっても、そのまま動く。
タイトル→READY→タイトルの遷移は、20年前の状態番号
(STEP_TITLE=0, STEP_READY=10)のまま新しい画面の上で回っている。

**「エンジンを替えてもゲームは変わらない」の最初の実証。**
今後 state.js・定義テーブル・ゲームロジックも同じ道を通る予定。

## 生まれた瞬間からフル装備

classic が14レッスンかけて獲得した装備を、新版は初日から着ている。

- ESLint / Prettier / 型検査の対象に組み込み(設定に2行足すだけ)
- JSDoc型は後付けではなく**書きながら**付けた
- E2Eテスト(タイトル表示・キー開始・タップ開始・canvas解像度)を
  器と同時に用意。「テストを書きにくい構造」になる前に書く
- デバッグ窓口 window.jibfreak.debug も classic の思想を踏襲

**新築の現場に、リフォームで学んだ全部を持ち込む。**
これがリファクタリングの本当の配当かもしれない。

## 教え

1. **エンジンは魔法ではない。** ループ・画面・入力・素材、
   分解すればどれも数十行。既製品を使うにしても、
   一度自分で書いた経験は「中で何が起きているか」の直感になる。

2. **速度は時間に掛ける。** フレーム基準の移動量は環境で
   速さが変わる古い設計。dt を受け取り、`px/秒 × dt` で動かす。

3. **移植で一番価値があるのは「変えなくていい部分」の発見。**
   flow.js の無修正移植は、レッスン14の設計が正しかったことの
   何よりの証拠。移植とは、依存の切れ目の答え合わせである。

## 放課後: 背景が地続きじゃなかった(そして先生の誤診)

公開後のプレイで「背景がループの継ぎ目で一度真っ暗になる」という
指摘を受けた。調べると bg.gif は **960px幅しかない**。classic の
定義には width: 1920 とあるが、DOM の background-image は
**勝手にタイル状に繰り返して**足りない分を埋めていた。
canvas の drawImage は言われた1枚しか描かない——エンジンを
替えたとき、「DOMが黙ってやってくれていた仕事」を1つ、
引き継ぎ忘れていたのだ。修正は2枚並べて描くだけ。

白状すると、この症状はもっと前からスクリーンショットに写っていた。
画面の途中に走る妙な縦の継ぎ目——それを先生(私)は「ヘッドレス
キャプチャのアーティファクト」と誤診して流した。目視で異変に
気づいたのは、実際に遊んだ人間のほうだった。
**自動テストは書いた分しか見ない。遊ぶ人の目は全部を見る。**
検証の最後のレイヤーは、いつだって実際に遊ぶことである。

## 放課後・特大回: GIFは動いていなかった

会長からの質問「素材はgifで用意しているが、canvasだとgifは
動かない?」——正解。canvas の drawImage はアニメGIFの
**1コマ目しか描かない**。DOM(classic)ではブラウザが勝手に
再生してくれていた。

そして全素材を調べて戦慄した。**33ファイルがアニメGIFだった。**
自機はプロペラが回り(2コマ)、雑魚は羽ばたき(2コマ)、
やられ演出は7コマの爆発で、猫バスは6コマで歩き、
タイトルの矢印カーソルまで2コマで点滅していた。
canvas移植後の JIB-FREAK MOBILE は、これら全てを
静止画で表示していた——「DOMが黙ってやってくれていた仕事」の
最大の見落としが、背景タイルに続いてまた見つかった形である。

### 対処: 1989年のフォーマットを自分で読む

選択肢は3つあった。ImageDecoder API(Firefox非対応)、
事前変換(no-build方針に反する)、そして**自前デコーダ**。
依存ゼロ・全ブラウザ動作・そして「エンジンを書く側に回った者」の
総仕上げとして、GIF89a仕様を読んでデコーダを書いた(約230行)。

- パーサ(engine/gif.js)は**DOMに依存しない純関数**として書き、
  Node の単体テストで実素材を検証できるようにした
  (ban.gif は7コマ、猫バスは6コマ——事前のバイト解析と一致)
- LZW展開、可変長コード、透明色、disposal(コマ間の後片付け)、
  インターレースまで実装。20年物の素材が相手でも、
  仕様書(1990年、W3C)が現役で通用する。**枯れたフォーマットは強い**
- アニメの時計はゲーム内時間に載せた。**ポーズすれば羽ばたきも
  止まる**——先日入れた「画面全体のピクセル静止」テストが、
  そのまま新機能の見張りになった

### テストが教えてくれたこと(今回も)

単体テストで「全コマに絵があるはず」と書いたら、ban.gif の
最終コマだけ真っ透明で落ちた。デコーダのバグを疑ったが、
正解は素材の内容——**爆発は消えて終わる**。7コマ目の「無」は
20年前のドット絵師(会長)の演出意図である。
テストの失敗は、コードの間違いか、思い込みの間違いかの
どちらかを教えてくれる。今回は後者だった。
