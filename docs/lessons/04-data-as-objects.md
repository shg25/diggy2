# レッスン04: 並行配列ではなく「1種類 = 1オブジェクト」

対応コミット: `288db7a`

## 何が問題だったか

1面の雑魚敵のパラメータは、こう書いてあった。

```js
angTeki   = new Array(rand±15, 0, rand±80, rand±45);
velTeki   = new Array(5, 10, 3, 8);
lifeTeki  = new Array(2, 4, 3, 2);
scoreTeki = new Array(50, 1000, 100, 250);
wTeki     = new Array(16, 16, 16, 16);
hTeki     = new Array(16, 16, 16, 16);
...
velocity : velTeki[n], angle : angTeki[n],
life : lifeTeki[n], score : scoreTeki[n],
```

「同じ添字で複数の配列を読む」——これは**並行配列(parallel arrays)**
というアンチパターン。データの持ち方が「縦」と「横」で逆になっている。

- 知りたいのは「**2番の敵**はどんな敵?」(横1行)
- 書いてあるのは「全敵の**速度一覧**」「全敵の**体力一覧**」(縦6列)

読むたびに6行を目で拾って頭の中で表を再構成することになる。
さらに悪いことに、5種類目の敵を足すとき**6本全部に要素を追加**しないと
いけない。1本忘れても、エラーは出ない。`velTeki[4]` は黙って
`undefined` を返し、敵は速度undefinedで生まれる。
レッスン03で見た「黙って壊れる」がここにもいる。

## 直し方

「1種類の敵 = 1つのオブジェクト」の定義テーブルにする。

```js
var TEKI1_DEFS = [
	{ width : 16, height : 16, velocity : 5,  life : 2, score : 50,   angRange : 30 },
	{ width : 16, height : 16, velocity : 10, life : 4, score : 1000, angRange : 0 },
	{ width : 16, height : 16, velocity : 3,  life : 3, score : 100,  angRange : 160 },
	{ width : 16, height : 16, velocity : 8,  life : 2, score : 250,  angRange : 90 }
];

var def = TEKI1_DEFS[n];
...
velocity : def.velocity, life : def.life, score : def.score,
```

1行 = 1種類。「1000点の敵は体力4で速い」が一目で読める。
種類を足すときは1行足すだけ。書き忘れる「別の配列」が存在しない。
同様に PWR_DEFS(アイテム3種)、BOSS_DEFS(ボス2体)、
BOSS_SH2_DEFS(2面ボス弾2種)も変換した。

## 挙動を変えないための工夫

このリファクタで一番頭を使うのは「本当に同じ動きか?」の検証。

1. **毎回ランダムな値は定義テーブルに入れられない。**
   進行角は湧くたびにランダムなので、値ではなく**ブレ幅**
   (`angRange`)を定義に持たせ、スポーン時に
   `Math.floor(Math.random() * angRange) - angRange / 2` で算出。
   元の式と同一の分布になる(ブレ幅0の敵は常に0度になるのも同じ)。

2. **元コードの「配列要素の書き換え」の意味を読む。**
   左から出る敵は `velTeki[n] = velTeki[n] * -0.4` と配列を
   書き換えていたが、配列は毎回作り直されるので持ち越しはない。
   つまり実質ローカル変数。だからローカル変数 `velocity` への
   代入で等価になる。もし配列がグローバルで持ち越されていたら
   話は別だった——レッスン03でスコープを整理しておいたから、
   この判断が一瞬でできた。**レッスンは積み重なる。**

## 教え

1. **データは「使うときの形」で持つ。** コードが `velTeki[n]` と
   `lifeTeki[n]` をいつも一緒に読むなら、それらは最初から
   一緒に置いておくべき。データ構造がコードの読みやすさを決める。

2. **「表」を書きたいなら、ちゃんと表に見える形で書く。**
   定義テーブルは縦に種類、横に属性が並ぶ。ゲームバランスの調整
   (このゲームで一番楽しい作業!)がテーブルの数字いじりになる。

3. **現代なら:** この定義テーブルはコードから出して JSON にする道もある。
   そこまでいくと「エンジン」と「データ」の分離になり、コードを
   触らずに敵を追加できる。いずれのレッスンで。

## 余談

検証プレイ中に、当時からのバグをひとつ発見した。敵数カウンタ
`numTeki` は初回プレイでは未初期化(undefined)のまま `numTeki++`
されて NaN になっている(リセット処理で 0 が入るのは2周目から)。
`NaN == 0` は false なので「敵が0なら必ず湧かせる」条件が初回は
一度も効いていない。挙動保存のため今回は温存。いずれ直す。
