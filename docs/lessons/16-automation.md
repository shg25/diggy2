# レッスン16: 自動化の門番 — git hooks と CI

対応コミット: `d5ada19`

## 「実行し忘れる検査」は存在しないのと同じ

レッスン12〜15で lint とテストが揃った。だが全部、
**人間が npm run と打てば**動くものだ。急いでいる日、疲れた夜、
「今回くらい大丈夫」の1回——検査は打たれずにコミットされる。
仕組みで防がない限り、必ず起きる(20年前の自分なら初日に起こす)。

そこで検査を2つの「門」に自動で組み込んだ。

## 門番1: pre-commit hook (ローカル・数秒)

git には hooks という仕組みがあり、コミットの直前に任意の
スクリプトを走らせて、失敗したら**コミット自体を中断**できる。

```sh
# .githooks/pre-commit
npm run lint && npm test
```

ここで走らせるのは lint と単体テスト(合計数秒)だけ。
E2E(80秒)は入れない——**遅い門番は必ず蹴破られる**からだ。
「--no-verify で飛ばす癖」がついたら門番は終わり。
コミットのリズムを壊さない速さが、守られる門番の条件。

ひとつ罠がある: hooks はリポジトリの外(.git/hooks)に住んでいて、
**クローンしても付いてこない**。そこで hooks をリポジトリ内の
.githooks/ に置き、npm install 時に走る prepare スクリプトで
`git config core.hooksPath .githooks` を実行させた。
これで新しく clone した人にも門番が自動で立つ
(husky という定番ツールの、依存ゼロ版)。

## 門番2: GitHub Actions CI (クラウド・数分)

.github/workflows/ci.yml を置くと、push や PR のたびに
GitHub のサーバーが仮想マシンを起動し、クリーンな環境で
検査を全部回してくれる。これが CI(継続的インテグレーション)。

```yaml
- run: npm ci        # 台帳どおりに依存を再現
- run: npm run lint
- run: npm test
- run: npm run test:e2e   # 重い検査はここで確実に
```

ローカルの門番との分担は「**速い検査はローカルで毎回、
遅い検査はクラウドで確実に**」。CI には副産物もある:
「まっさらなマシンで npm ci から動く」ことの証明だ。
自分のマシンでしか動かない状態(works on my machine)を
毎push検出できる。

レッスン15のE2Eが「サーバー起動から後片付けまで自己完結」に
作ってあったのが、ここで効いた。CI用の特別な準備なしで
そのまま動いた。

## 教え

1. **規律を意志力で維持しない。仕組みにする。**
   「毎回テストを打つ」という決意は必ず破れる。
   打たなくても走る形にすれば、破れようがない。

2. **門番は速さで配置する。** ローカル=秒、CI=分。
   逆にすると、ローカルの門番は飛ばされ、CIは待たれなくなる。

3. **CIは「他人の環境で動く」ことの継続的な証明。**
   このリポジトリを20年後にまた開くとき、CI が緑なら
   「少なくとも Actions の環境では全部動く」ことが分かる。
   今回の発掘作業(レッスン01)で一番欲しかった情報だ。

## 余談

フォーマッタ(Prettier)は今回見送った。導入自体は簡単だが、
全ファイルを一斉に再整形する大きなコミットを伴い、このコードの
「当時の手触り」(`key : value` のスペースなど)も消える。
機能ではなく見た目の全面書き換えをどう扱うかは、それ自体が
ひとつの意思決定なので、別途相談して決める。
