# レッスン22: 移植その1 — 変わるものと変わらないもの

対応コミット: `f11e72c`

## 移植は3種類の作業でできている

自機とショットを classic から canvas エンジンへ移植した。
やってみると、移植とは1種類の作業ではなく、
**3種類の作業の混合**だと分かる。

### 1. 無修正で運べるもの(コピー)

state.js、const.js は**1文字も変えずに**コピーした(flow.jsに続き3つ目)。
定義テーブル(JIKI_SH_DEFS)も値は完全にそのまま。
これらに共通するのは「エンジンを知らない」こと。
純粋なデータとロジックは、下の土台が DGE だろうが canvas だろうが
関係なく動く。**レッスン14で下から切り離したものが、そのまま
持ち上がった。**

### 2. アルゴリズムは同じ、器だけ替わるもの(書き写し)

ショットのプール選択・レーザーの3方向展開・発射位置の計算は、
classic の set.js のアルゴリズムをそのまま書き写した。
変わったのは `jikiSh1[i].get('active')` が `jikiSh1[i].active` に
なった、という**器の API だけ**。ロジックの形は完全に保存されている。
20年前に考えた「3発同時が全部空いていたらレーザーを撃つ」という
判断は、canvas の上でも同じ判断のまま生きている。

### 3. 仕様を調べて再実装するもの(考古学)

一番手間だったのがここ。定義テーブルの `angle: 180` が
「どっちを向いているか」は、Diggy のソースを読まないと分からない。

```js
// lib/diggy/core/dge.js より
DGE.getCoordsByAngleVelocity = function (angle, velocity) {
	angle = 270 - angle; // ← 独特の読み替え
	...
};
```

Diggy の角度は `270 - angle` してから sin/cos に掛かる独自規約
だった(angle 180 = 右向き)。この規約を entity.js に互換再実装する
ことで、テーブルの角度をそのまま使える。
ちなみにこの関数の直前には作者の魂の叫びが残っている:

> // What the fuck is wrong with me I used to know how to do this

20年前のエンジン作者も、角度の変換で頭を抱えていた。
**仕様書のないコードでは、実装だけが仕様である。**
だから借り物でも、いざという時に読めるコードを選ぶ価値がある。

## 単位の境界を1箇所に集める

classic の速度は「px/フレーム(30fps基準)」、新エンジンは
「px/秒 × dt」。この換算をあちこちに書くと必ず事故るので、
**entity.js の advance() 1箇所だけ**が換算(`velocity × FPS × dt`)を
知っている。定義テーブルは当時の単位のまま触らない。
境界の変換は境界に置く——型のときの d.ts(レッスン18)と同じ発想。

## テストが移植の答え合わせになる

E2E を新しい遷移に合わせて書き直した:
タイトル→READY(2秒)→START、矢印で移動して**画面端で止まる**、
スペースで撃てる、Z でレーザーに切り替えると**同時に3発出る**。
「classic と同じ手触りか」を人間の記憶だけでなく、
数字(x座標、弾数)でも確かめられる。

## 教え

1. **移植の工数は「コピー・書き写し・考古学」の比率で決まる。**
   見積もる前に、どの部分がどの種類かを仕分けする。
   今回はレッスン14までの下ごしらえのおかげで、
   考古学が1件(角度規約)で済んだ。

2. **互換をとるなら、変換は1箇所に。** 単位や座標系の違いを
   コードのあちこちで吸収し始めたら、もう誰にも追えない。

3. **古いコードの独自規約は、批判せずにまず再現する。**
   「270-angle は変だ」と直したくなるが、直すと定義テーブル全部の
   角度を書き換える羽目になり、移植とリファクタが混ざる(レッスン02
   の「動かす変更と変える変更を混ぜない」)。まず動かす。直すのは後。

## 余談: テストがまた設計の性質を教えてくれた

E2E の初回実行で「Z 2連打でレーザーに切替」が落ちた。原因は
新エンジンの入力設計: 「このフレームで押されたキー」を Set で
持つため、**同一フレーム(16ms)内の同キー2連打は1回に潰れる**。
人間には16msで2回押せないので実害はないが、テスト(機械)は
押せてしまう。テスト側で押下間隔を空けて解決。
機械だけが踏める境界条件を、機械のテストが見つけてくる。

ついでに白状すると、この失敗を最初「ALL PASS」と誤報した。
検査コマンドを `npm run test:e2e | tail` とパイプしていたため、
終了コードが tail のもの(成功)にすり替わっていた。
シェルのパイプは最後のコマンドの終了コードを返す——
以後は pipefail を付ける。**検査の報告経路そのものも検査対象。**

## 放課後: 押しっぱなしで連射

classic は1発ごとにスペースを押す必要があった(20年前の
指が覚えている連打)。新版では押しっぱなしで連射するよう変更。
実装は「押した瞬間(wasPressed)」に加えて「押している間(isDown)」
でも発射するだけ——**弾数の制限はもともとプール(3発)が
守っている**ので、発射側は無心に引き金を引いてよい。
制限をトリガー側ではなく資源側(プール)に置いていた
20年前の設計が、ここでまた配当を生んだ。
なお classic は当時の連打の手触りごと凍結保存されている。

なお公開後、押しっぱなし連射は「3発が密着して出て近距離が
強すぎる」ことが判明し、第5回生徒会(seitokai/05)でホールド連射に
間隔(0.15秒)が入った。新規の押下は即発射なので、最速の連射は
今も物理連打で出る——20年前の連打の腕前は、無駄にならなかった。
