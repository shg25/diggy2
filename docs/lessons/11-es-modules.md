# レッスン11: 部屋に仕切りを — ES modules への移行

対応コミット: `fdb0a63`

## そもそも: なぜ script タグが20本並んでいたのか

当時のJavaScriptには「ファイルを分ける公式な仕組み」が**なかった**。
複数ファイルに分けるには、scriptタグを正しい順番で並べ、
ファイル間の受け渡しはすべてグローバル変数で行うしかなかった。

```html
<script src="js/const.js"></script>   <!-- これが先じゃないと -->
<script src="js/sprite.js"></script>  <!-- こっちが壊れる -->
...
```

レッスン03の「暗黙のグローバル27個」も、繰り返し出てきた
「誰がどこで書き換えるか分からない変数」も、根本原因はこれ。
**言語に部屋の仕切りがなかった**。皆が同じ大部屋(グローバル空間)で
生活していた。

2015年、ES2015でようやく公式のモジュール機構 **ES modules** が入った。

```js
// sprite.js — export したものだけが外から見える
export function makeTeki1() { ... }

// stage.js — import で受け取る
import { makeTeki1 } from './sprite.js';
```

- ファイルごとに独立したスコープ。export しない限り外から見えない
- 依存関係がファイル冒頭に**宣言**される。scriptタグの並び順パズルは
  ブラウザが import を辿って解決する
- 自動的にstrictモード(レッスン03で手書きした 'use strict' は不要に)

index.html のゲーム側は、エントリポイント1行になった。

```html
<script type="module" src="js/ridge.js"></script>
```

エンジン(lib/diggy/)は借り物なので触らず、従来の script タグのまま。
`DGE` というグローバルをゲーム側が参照する形で共存できる。

## 最大の難所: モジュールをまたぐ「書き換え」

import した変数は**読み取り専用**だ。

```js
import { velJiki } from './sprite.js';
velJiki = 10;   // TypeError!
```

読むだけなら import で足りる(live bindingといって、元が変われば
読む側にも反映される)。だが RIDGE部には「set.js が velJiki を
書き換え、stage.js が読む」のような**モジュールをまたぐ書き換え**が
7つあった。これらは `state.js` という1ファイルに集約した。

```js
export const state = {
	counter : 0,    // ステージ進行のフレームカウンタ
	stageFlg : 1,   // 選択中のステージ
	velJiki : 5,    // 自機の移動速度
	...
};
```

オブジェクトのプロパティ書き換え(`state.velJiki = 10`)は許される
(読み取り専用なのは binding であって中身ではない)。

これは制約への回避策ではなく**収穫**だ。「このゲームで複数の場所から
書き換えられる状態は、この7個で全部」という事実が、1ファイルを
開くだけで分かるようになった。モジュール化以前、この7個は
100個近いグローバルの海に沈んでいて、探し出すこと自体が仕事だった。

## もうひとつの収穫: 出入口が設計になった

各ファイルの冒頭を見ると、そのファイルが「何に依存し(import)、
何を提供するか(export)」が宣言として読める。たとえば
text.js は何も import しない(=一番下の層)、ridge.js は全部を
import する(=一番上の入口)、という**階層構造**が初めて可視化された。

副作用として、検証スクリプトがコンソールから内部を触れなくなった
(レッスン03の余談で「チートには無防備」と書いた、あの遊びが
できなくなった)。代わりに `window.ridge.debug` という
**意図して公開する唯一の窓口**を entry に定義した。
「全部見える」から「見せると決めたものだけ見える」へ。
これがカプセル化という言葉の実体だ。

## 教え

1. **グローバル変数は言語の欠落への対処だった。** 当時の書き方が
   下手だったのではなく、道具がなかった。道具が来たら乗り換える。

2. **「書き換えられる共有状態」はプログラムで一番危険な部分。**
   モジュール化はそれを消してはくれないが、**一覧にして見せて**
   くれる。見えるものは管理できる。

3. **公開は最小に、そして意図的に。** export を書くたびに
   「これは本当に外に見せるものか?」と一度考える。
   窓口が狭いほど、中を自由に変えられる。

## 余談

移行中いちばん頭を使ったのは循環参照だった。sprite.js は step.js の
goLose を呼び、step.js は sprite.js の newSpriteBoss を呼ぶ。
ES modules は関数どうしの循環なら問題なく扱える(関数宣言は評価前に
確定するため)が、モジュール読み込み時にお互いの変数を読み合うと
壊れる。今回は全ファイルの初期化コードが自己完結していたので
無事だった。循環自体は設計の臭いでもあるので、いずれ
「レイヤー分け」のレッスンで解消したい。
