# レッスン23: 移植その2 — 敵と当たり判定、そして契約の是正

対応コミット: `540230a`

## 平和の終わり

雑魚敵4種・弾との当たり判定・スコア・被弾からの GAME OVER を
移植した。JIB-FREAK MOBILE は「撃ち放題の射撃場」から
「死ねるゲーム」になった。ゲームは死ねてはじめてゲームである。

## 物理は30Hzの固定tickで回す

今回いちばん大事な設計判断。新エンジンのループは
requestAnimationFrame で、60Hz にも 120Hz にもなる。
しかし classic のゲームバランスの数字は、すべて
**「30fpsのフレームごと」に調整されている**。

- 湧き確率「毎フレーム 1/20」→ 60Hzで判定したら敵が2倍湧く
- 端での跳ね返り「angle × -0.7」→ 60Hzなら2回掛かって別の軌道になる

そこで敵の物理・判定・湧きは、dt を積み立てて
**1/30秒たまるごとに1回**処理する(固定タイムステップ)。

```js
tickAccum += dt;
while (tickAccum >= 1 / FPS) {
	tickAccum -= 1 / FPS;
	tick(); // classicの1フレーム分
}
```

描画は毎フレーム(なめらか)、ゲームの判断は30Hz(当時のまま)。
**「見た目の滑らかさ」と「挙動の忠実さ」を別のレートで両立**させる、
移植の定石。

## 20年前の負債を、新築で返す

classic の numTeki には「減算は敵自身、加算は呼び出し側」という
非対称な契約があり、テストがその罠を踏んだ(レッスン15の余談)。
classic 側は挙動保存のため温存したが、**新築では直す**。
新版の spawnTeki1() は生成と同時に自分で数える。
対称な契約は、呼び出し側(将来のステージ2、テスト、ボス)が
何も知らなくても正しく動く。

移植は「同じものを作り直す」作業ではない。
**持ち越すものと、置いていくものを選ぶ**作業だ。
ゲームバランスの数字は1つ残らず持ち越し、歪んだ契約は置いてきた。

## 依存性逆転は、エンジンが替わっても同じ形

被弾処理はこう書いてある。

```js
// enemies.js (下位)
transitions.lose(); // 「当たった」と知らせるだけ

// main.js (上位)
transitions.lose = () => { setStep(STEP_LOSE); killJiki(); ... };
```

レッスン14で classic に入れた transitions フックが、
canvas エンジンの上でも同じ形で機能している。
敵は GAME OVER の演出を知らないまま、プレイヤーを倒せる。

## 教え

1. **バランスの数字には「単位」がある。** 1/20 という確率は
   「30fpsのフレームあたり」という暗黙の単位付きだった。
   数字を移植するときは、値だけでなく単位(いつ、何回評価されるか)
   ごと移植する。固定タイムステップはそのための道具。

2. **移植は返済のチャンス。** 挙動保存の縛りで直せなかった
   歪みは、新しいコードでは最初から正しく書ける。
   ただし何を「歪み」と判断したかは記録する(このファイルがそれ)。

3. **ゲームオーバーまで作って、はじめて1周。** 敵だけ居ても
   ゲームではない。死→GAME OVER→タイトル復帰の環が閉じて、
   ようやく「遊べる」と言える。機能は環で数える。

## 余談

E2E に「敵に当たると GAME OVER になる」というテストを足した。
やっていることは「無敵を切って、死ぬまで待つ」。
死ぬことを成功条件にするテストというのも妙なものだが、
シューティングゲームにとって、ちゃんと死ねることは
ちゃんと遊べることと同じくらい大事な仕様である。
