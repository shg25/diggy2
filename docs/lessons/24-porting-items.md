# レッスン24: 移植その3 — アイテムとボム、直したバグは移植しない

対応コミット: (TBD)

## 3つのご褒美

パワーアップアイテム3種を移植した。取ると
[0] ショット変更(前後撃ち→レーザーと巡る)、[1] スピード変更、
[2] ボム(一定時間、雑魚が触れただけで消える)。
classic では B/Z/S の隠しコマンドと同じ効果関数を呼ぶ作りに
なっていて、その「アイテムとチートが同じ関数を共有する」構造ごと
移植した。20年前の自分は、効果のロジックを1箇所に置くことを
(たぶん無自覚に)できていた。

## 直したバグは移植しない

classic のアイテムには「画面外に消えるとき敵数カウンタ numTeki を
減らしてしまう」バグがあった(レッスン05で修正済み)。
移植元は**修正後の classic** なので、新版には最初からバグがない——
当たり前のようだが、これはレッスン05の修正が
「その場しのぎ」ではなく「系譜に効く直し方」だったことの証明だ。
もしバグを放置したまま移植していたら、バグごとコピーされ、
2つのゲームで同じ虫を2回退治する羽目になっていた。

**バグは見つけた世代で殺す。次の世代に相続させない。**

E2E には「アイテムが湧いたあとも numTeki と実際の敵数が一致する」
というテストを足した。レッスン05の再発防止が、新しいエンジンにも
引き継がれた形になる。

## tick の置き場所を classic の形に揃える

前レッスンで敵モジュールに置いた30Hzの蓄積器を、stage.js に
移した。classic の stage.js(stage1 interval)が「counter を進め、
アイテムを湧かせ、敵を湧かせる」の束ね役だったのと同じ配置になる。

```js
// stage.js — 30Hzごとに「1フレーム分の判断」を束ねる
state.counter += 1; // ボス登場のタイミングに使う(次レッスンの布石)
tickPwrs();
tickTekis();
```

移植先の構造を移植元に**似せておく**と、次に運ぶもの(ボスは
counter を見て登場する)の置き場所が自動的に決まる。
地図が同じなら、迷わない。

## 教え

1. **修正済みのコードから移植する。** 移植元の選び方はそれ自体が
   意思決定。「動いていた最古のバージョン」ではなく
   「一番賢くなった最新の姿」から運ぶ。

2. **効果とトリガーを分ける。** ボムの効果(rmGroupTeki)は1つ、
   トリガーはアイテム取得とBキーの2つ。効果関数を共有していれば、
   トリガーがいくつ増えても効果は1箇所で直せる。
   (20年前の自分がすでにやっていた設計。良い設計は時代を越える)

3. **移植は構造も運ぶ。** 関数だけ運んで置き場所を変えると、
   元のコードとの対応が取れなくなり、次の移植で迷子になる。
   「classic の stage.js に相当するのは新版の stage.js」という
   1:1の地図を保つ。

## 余談

ボムのE2Eテストは「敵が2体以上たまるのを待って、Bを押して、
スコアが入ることを確かめる」。書いていて気づいたが、これは
20年前に友人とやっていた動作確認そのものだ。「ボムためして」
「敵くるまで待って」「おっ消えた消えた」——あの会話が
テストコードという形で、毎回自動で再演されている。
