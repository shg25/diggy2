# レッスン05: 数えるなら対で数える — NaNになったカウンタ

対応コミット: `d8855f6`

## 何が問題だったか

レッスン04の検証プレイ中に見つけた、当時からのバグを直す回。
敵数カウンタ `numTeki` に2つの問題があった。

### 問題1: 初期値がない

```js
var numTeki;          // undefined のまま
...
numTeki++;            // undefined + 1 = NaN
```

`numTeki` は宣言だけで初期値がなく、初回プレイで `undefined` のまま
`++` されて **NaN** になっていた。0 が入るのはゲームオーバー後の
リセット処理が走る2周目から。

これの何が困るか。1面には「敵が0体なら必ず湧かせる」という
補給ロジックがある。

```js
if ((numTeki == 0) || (DGE.rand(1, 20) == 1)) {
```

`NaN == 0` は false。NaN はどんな比較も false になるので、
**初回プレイだけこの条件が一度も効かず**、敵の湧きが1/20の運任せに
なっていた。1周目と2周目で難易度が微妙に違うゲームだったのだ。
誰も気づかないまま20年経った。

### 問題2: 数えていないものを減らしていた

パワーアップアイテムが画面外に消えるとき、`numTeki--` していた。
アイテムは敵数に一度もカウントインされていないのに、である。
おそらく雑魚敵のイベントハンドラをコピペしてアイテムを作ったとき、
`numTeki--` を消し忘れた。**コピペは「余計なものまで運ぶ」**。

NaN には何を足しても引いても NaN なので、皮肉なことに問題1が
問題2を隠していた(2周目以降は実害があった: アイテムを取り逃すたびに
カウンタがずれ、湧き判定が狂う)。

## 直し方

「増やす場所」と「減らす場所」を対にする、それだけ。

1. `var numTeki = 0;` — 宣言と同時に初期化。
2. アイテムの画面外処理から `numTeki--` を削除。

なお2面の敵(とぅと郎)を撃破したときも `numTeki--` される非対称が
残っている(カウントインされずに減る)が、2面は numTeki を一切
読まないため実害がなく、今回は見送った。

## 教え

1. **カウンタは資源管理。** 増やす場所と減らす場所を紙に書き出して
   対応が取れているか確認する。取れていなければどちらかがバグ。

2. **NaN は伝染して、比較をすべて false にする。**
   「エラーにならないのに何かおかしい」ときは NaN を疑う。
   現代なら `Number.isNaN()` でのガードや、そもそも
   undefined になり得ない初期化を。

3. **バグ修正とリファクタは別コミット。**
   このコミットは挙動を「変える」(初回プレイの湧きが正しくなる)。
   レッスン03〜04は挙動を「変えない」。この区別を git 履歴に
   残しておくと、将来「どこから挙動が変わった?」を探すとき
   (git bisect)に効く。

## 余談

このバグの発見経緯が面白い。レッスン04の検証で内部状態をダンプしたら
`"numTeki": null` と出た。JSON には NaN が存在しないので null に
変換されて表示されたのだ。「あるはずの数字が null」という違和感から
逆算して、初期化漏れに辿り着いた。**検証は書いたコードの確認だけで
なく、書いてないコードのバグも網にかける。**
