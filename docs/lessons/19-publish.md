## GitHub Pages とは

リポジトリの中身を、そのまま静的Webサイトとして配信してくれる
GitHub の機能。無料プランではパブリックリポジトリで使える。
今回は「main ブランチのルートを配信する」最小構成にした。

```
リポジトリの index.html
  → https://shg25.github.io/diggy2/ として世界に配信
```

以後、main に push するたびに本番も自動で更新される。
デプロイ作業という概念が存在しない。

## 過去の自分たちへの答え合わせ

公開作業は設定2つで終わった。それは偶然ではなく、
これまでのレッスンの積み立てが満期を迎えた瞬間だった。

- **レッスン02**: index.html をルートに昇格させた構成が、
  Pages の「ルートを配信」にそのまま一致
- **レッスン18**: 「ビルドしないTypeScript」を選んだので、
  ビルド工程もデプロイ用の変換も一切不要。ファイルがそのまま本番
- **ライセンス確認(README)**: エンジンがBSDで同梱LICENSEが
  条件を満たすことを確認済みだったので、公開判断が一瞬
- **favicon解消・README整備**: 初見の訪問者への身だしなみが済んでいた

**公開は「最後の大仕事」ではなく「日々の積み立ての清算」**に
なるのが理想で、今回はそれができた。

## パブリック化の意味

リポジトリを公開すると、コードだけでなく**全コミット履歴・
レッスン文書・Issue・CIの実行結果**まで誰でも見られる。
今回は困るものが何もないことを確認済みだったが、原則として:

- 秘密情報(APIキー等)は一度でもコミットしたら「削除しても
  履歴に残る」。公開前に履歴ごと確認する
- 公開は事実上不可逆と考える(クローン・魚拓は止められない)

## 本番でも「動くはず」ではなく「動いた」を取る

公開直後、本番URLに対してスモークテストを流した。
ヘッドレスChromeで本番ページを開き、タイトル→開始→プレイ、
リソース404ゼロ・実行時エラーゼロを確認。

結果が面白かった。最終状態が STEP_LOSE——テスト用の無敵化を
していないので、**敵に当たって普通にゲームオーバーになった**。
アサーションとしては失敗扱いだが、実証としては満点である。
「本番で、ゲームとして、ちゃんと殺してくる」。

ローカルで動くことと本番で動くことは別(パスの違い、HTTPSの制約、
CDNのキャッシュ…)。環境が変わったら検証もやり直す。

## 教え

1. **公開可能な状態を保ち続けると、公開はイベントでなくなる。**
   「いつでも出せる」は品質の総合指標。逆に「公開前に
   やることリスト」が長いプロジェクトは、日々の負債を
   公開日に一括返済しているだけ。

2. **リポジトリ設定の変更は git に残らない。** パブリック化も
   Pages 有効化も、コミット履歴のどこにもない。だからこうして
   文書に残す。将来は設定をコード化(Actions によるデプロイ
   ワークフロー等)して履歴に乗せる手もある——2本立て構成に
   組み替えるときの課題。

3. **URLを持つと開発が変わる。** 「友人に送れる」「スマホで開ける」
   は、ローカルで完結していた頃には存在しなかった開発の燃料。
   以後の機能追加は、push した瞬間に全員の手元で更新される。
