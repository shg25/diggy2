## 状態は持ち主のところへ

このステージの移植では、classic の「共有状態の warts」を
2つ、正しい持ち主に返した。

1. **猫バスの暴れ状態(bossTurn / bossTurnMode)**
   classic ではファイルレベルの変数だった。暴れているのはボスなのだから、
   状態はボスのフィールド(boss.turnDir / boss.turnMode)に持たせる。
2. **追尾弾の状態(turn)**
   classic では**全弾で1つのグローバル**を共有していて、弾が2発同時に
   存在すると状態を奪い合った(レッスン03で発見した wart)。
   弾ごとのフィールド(shot.homing)にした。

どちらも「classic 側は挙動保存のため温存し、新築では最初から正しく」
という、この移植シリーズで繰り返してきた判断の総仕上げ。

## 数字で振り返る移植プロジェクト(レッスン21〜26)

- classic から**無修正で運べたファイル**: flow.js, state.js, const.js(3本)
- **値を1つも変えずに運んだ定義テーブル**: 5つ(自機ショット・雑魚・
  アイテム・ボス・ボス弾)
- **考古学が必要だった仕様**: 1件(Diggy の角度規約 270-angle)
- **新築で是正した warts**: 3件(numTeki の非対称契約、暴れ状態の
  グローバル、追尾弾の共有状態)
- **classic に無かったものを配線**: 1件(勝利)
- エンジン本体: 約150行 + エンティティ共通処理

移植を通して一番効いたのは、レッスン14のレイヤー分けだった。
「下から切り離してあったものはそのまま持ち上がり、
絡んでいたものだけ書き直しになる」——移植の工数は、
移植を始めるずっと前の設計で決まっていた。

## 教え

1. **移植は設計の成績表。** きれいに運べた部分は疎結合だった部分、
   書き直しになった部分は密結合だった部分。次の設計への
   フィードバックとして、これほど正直な資料はない。

2. **状態には持ち主がいる。** 「ボスの状態」はボスに、「弾の状態」は
   弾に。グローバルに置かれた状態は、持ち主を探している迷子。

3. **移植の終わりは、比較の始まり。** 2つの実装が同じ遊びを
   提供している今、classic と jibfreak を並べて遊べば、
   エンジンの違い(30fps DOM vs 60fps canvas)が手触りとして分かる。
   同じ設計図から建てた2軒の家は、最高の教材になる。
