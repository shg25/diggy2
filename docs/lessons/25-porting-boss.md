## 発掘調査で見つかった衝撃の事実

ボスを移植する前に、classic の勝利まわりの配線を確認した。
goWin は完全に作り込まれている——「YOU WIN」の表示、5秒の余韻、
後片付け、タイトルへの帰還。ところが:

```sh
$ grep -rn "goWin" classic/js/
flow.js:   ログ出力の分岐
ridge.js:  デバッグ窓口への公開
step.js:   定義
```

**呼び出しがどこにもない。** ボスを倒すと5000点入って消えるが、
勝利判定は発火せず、ステージ進行は回り続け、ボスの亡骸の座標から
弾幕が撃ち続けられる。つまり RIDGE部は、
**20年間、誰も勝てないゲームだった**。

演出を作り込んだのに配線を忘れる——おそらく「あとで繋ぐ」つもりの
最後の1本だったのだろう。テストがあれば initial commit の前に
気づけたはずで(「ボスを倒すと勝てる」は最初に書くべきテストだ)、
これはレッスン15「検証を資産に」のもっとも痛烈な実例になった。
ちなみに検証していた私たちも気づかなかった。E2E がデバッグ窓口から
goWin() を直接呼んでいたからだ。**テストが実装を経由しないと、
配線の断線は検出できない。**

## 新版では意図どおりに配線する

flow.js の transitions に win フックを追加し(classic コピーからの
初の意図的な divergence)、ボス撃破 → transitions.win() → YOU WIN
5秒 → タイトル、という当時設計された環をついに閉じた。

E2E のテスト名はこうした:
**「ボスを倒すと YOU WIN、5秒後にタイトルへ戻る(20年越しの初勝利)」**。
このテストはデバッグ窓口で goWin を呼ばない。ボスを瀕死にして
射線上へ引きずり出し、**正規の当たり判定で撃破する**。
断線の教訓から、勝利への配線そのものを試験対象にした。

## 移植メモ

- ボス本体・弾幕(交互の砲門から角度を変えて撃つ)・レーザー・
  「後方ショットは当たらない」「レーザーは0.2倍」の仕様は
  classic の値のまま
- ボム(rmGroupTeki)がボス弾も消す仕様も移植
- 負けたときにボスが飛び去る getoutBoss も移植
- 同フレーム多段ヒットで transitions.win() が2回発火しうる穴を
  移植時に発見してガード(classic は「勝利がない」ので、
  この穴も存在すらしなかった。配線すると穴が生まれる)
- GO!! の1秒後にステージ進行が始まる classic のタイミングも再現

## 教え

1. **「最後に繋ぐ」は繋がらない。** 部品を全部作ってから配線する
   進め方は、最後の1本を忘れる。細くてもいいから先に環を通し
   (トレーサー弾)、部品はあとから太らせる。

2. **テストは実装の経路を通す。** ショートカット(デバッグ呼び出し)で
   検証した機能は、本物の経路が断線していても緑になる。
   便利な裏口は、検証では表玄関の代わりにならない。

3. **機能を足すと、無かった穴も生まれる。** 勝利を配線した瞬間、
   「二重勝利」という classic には存在しえなかったバグの芽が生じた。
   変更の影響は「足したもの」だけでなく「足したものが作る新しい
   状態」まで見る。

## 余談: ゲームが強くなると、テストが死ぬ

ボスを入れた途端、既存の「矢印キーで移動できる」テストが落ちた。
原因はテストコードでもボスのバグでもなく、**テスト中の自機が
弾幕に撃たれて死んでいた**。そのテストは無敵化しておらず、
ボスという新しい死因が増えたことで寿命が尽きたのだ。
移動テストの目的は移動の検証であって生存ではないので、無敵化を
追加して解決。「テストは検証したいこと以外の条件を固定する」
という基本を、ボスが体を張って教えてくれた。
