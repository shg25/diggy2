# レッスン20: 引っ越しと看板 — 2バージョン公開の構成

対応コミット: `f61cd86`

## 決めたこと

このプロジェクトは今後、2つのゲームを公開する。

- **RIDGE部**(旧作): ゲームは当時のまま、コードだけ現代化した保存版
- **JIB-FREAK MOBILE**(新版): 同じ素材で、エンジンから作り直し、
  新機能を積んで進化していく版。名前は当時2人が働いていた
  会社の空気を受け継いだもの(英語圏スラングの確認済み。
  "FREAK" はポケモンの Game Freak と同じ、ゲーム文化では
  市民権のある語)

RIDGE部という名前は友人があまり気に入っていなかった——
という20年越しの記憶に、新しい名前で応える。

なお、最初の案は会社の造語をそのまま使った "FREEK" だったが、
公開直後に「もじりではなく社名そのものに近すぎる」と気づいて
一般綴りの FREAK に改めた。**名前は公開してから間違いに気づく**。
だから早めに看板を出して、直せるうちに直すのが正しい。

## 構成: モノレポ + ディレクトリ + 自己完結

```
/              トップページ(看板)。両ゲームへの入口
/classic/      RIDGE部一式(index.html, js/, gfx/, lib/, types/)
/jibfreak/     JIB-FREAK MOBILE(これから)
/docs/ /test/  リポジトリ共通の記録とテスト
```

検討した選択肢と、選んだ理由:

1. **タグやブランチから配信を組む** 案は見送り。仕組みは美しいが、
   「見ればわかる」度でディレクトリに負ける
2. **素材(gfx)の共有はしない**。数十KBのGIFなので各版がコピーを持つ。
   共有すると「新版のために素材を触ったら旧作も変わった」という
   事故の芽になる。**旧作の凍結保存 > DRY**。重複が正解の場面もある
   (レッスン06「たまたま似ているだけの別物はまとめない」の応用——
   今後は意図的に別々の道を歩むのだから、始めから別物)
3. **開発ツール(lint/型/テスト/CI)はルートで共通**。守りの仕組みは
   全バージョンに同じ品質で効かせたい

## 引っ越しの作法

移動はすべて `git mv`(70ファイル、全部 rename 検出 100%)。
レッスン02でやった引っ越しと同じ作法だが、今回は追従する相手が
増えていた: eslint の対象パス、tsconfig の include、
prettierignore、テストの import と URL、README の構成図。

**コードを動かすと、コードを指している全員が動く。**
これを漏れなく追従できたのは、指している場所が設定ファイルに
明示されていたからで、これも現代化の配当。移動後に全門番
(format/lint/型/単体/E2E)を回して実証した。

## 新版の技術方針(予告)

JIB-FREAK MOBILE は Diggy エンジンを使わない。
**フレームワークなしの canvas + 自作ミニエンジン**でいく。

- Diggy の作者自身が README に「やっぱり canvas でゲームを
  作りたくなったから捨てる」と書き残している。その道を
  20年後に自分たちで歩く
- レイヤー分け(レッスン14)の配当がここで最大化する。
  flow.js(状態機械)・state.js・定義テーブル・ゲームロジックは
  ほぼそのまま移植でき、差し替わるのは最下層の「描画とループ」
  だけのはず——「エンジンを替えてもゲームは変わらない」を
  アーキテクチャで証明する実験でもある
- 「借り物は触らない」でやってきた私たちが、今度は
  エンジンを書く側に回る

## 教え

1. **凍結したいものと進化させたいものを、同じ場所に置かない。**
   保存版と開発版の同居は、どちらかの目的を必ず裏切る。
   物理的に分ければ、片方を壊す心配なしにもう片方で暴れられる。

2. **共有は契約。** ファイルを共有した瞬間、両利用者は
   「勝手に変えない」という暗黙の契約で縛り合う。
   その契約を守る自信がないなら、コピーの方が安い。

3. **名前は当事者の気持ちが最優先。** 技術的にどれだけ正しくても、
   作った本人たちが愛着を持てない名前のプロダクトは育たない。
   (ただし公開するなら他言語での意味チェックだけは怠らない)

## 余談

トップページの見出しは「SHOOTING GAMES」と複数形にした。
2つ目のゲームはまだ COMING SOON の札しかないが、
複数形にした瞬間、このプロジェクトは「ゲームの保存庫」から
「ゲームスタジオ」になった気がする。友人に見せるとき、
この札があるのとないのとでは、話の弾み方が違うはずだ。
