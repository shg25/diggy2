# レッスン03: 変数は宣言してから使う — 暗黙のグローバルという時限爆弾

対応コミット: `5aef35d`

## 何が問題だったか

JavaScriptでは、`var` を付けずに変数へ代入すると、エラーにならず
**黙ってグローバル変数が作られる**。当時のコードはこれだらけで、
宣言なしの変数が27個あった(sprite.jsに26個、text.jsのtxtStageで1個)。

```js
function makeTeki1() {
	velTeki = new Array(5, 10, 3, 8);   // グローバル!
	n = Math.floor(Math.random() * 4);  // グローバル!
	...
}
```

「動いてるからいいじゃん」と当時は思っただろう。実際に何が起きていたか。

### 実害1: 別の関数のゴミを読んでいた

1面ボスの弾を作る `makeBossSh1` には、こう書いてあった。

```js
new DGE.Sprite({
	...
	n : n, group : groupBossSh   // この n、どこにも代入してない
})
```

この関数のどこにも `n` への代入がない。読んでいたのは、**直前に
たまたま湧いた雑魚敵やアイテムが、グローバル `n` に残していったゴミ**。
壊れなかったのは、ボス弾の `n` プロパティを誰も読まなかったという
二重の偶然にすぎない。今回この2箇所は削除した。

### 実害2: 状態の混線

2面ボスの追尾弾は、発射から一定時間後に自機へ向き直る。その
「向き直ったかどうか」を管理する `turn` がグローバルだったので、
**追尾弾が2発同時に存在すると、1つのフラグを2発が奪い合う**。
弾ごとに持つべき状態が、全弾で共有されていた。

## 直し方

原則はひとつ: **変数は、それが必要な最小のスコープで宣言する。**

1. その関数の中でしか使わない → 関数内で `var`
   (`velTeki`, `n`, `lr`, `angBoss` など大半はこれ)
2. フレームや関数呼び出しをまたいで生きる必要がある →
   ファイルの先頭で `var` (ボスの暴れ状態 `bossTurn`,
   `bossTurnMode`, 敵の使い回し `teki2` など)

そして再発防止に、全ファイルの先頭へ `'use strict';` を追加した。
strictモードでは宣言なしの代入が**その場でReferenceErrorになる**ので、
この種のバグは書いた瞬間に発見される。

なお `turn` は「本来は弾ごとに持つべき」だが、このレッスンでは
挙動を変えないため、あえてファイルレベルの共有のまま宣言だけした。
構造の修正は別のレッスンでやる。**「宣言を正す」と「設計を直す」を
同じコミットに混ぜない。**

## 教え

1. **エラーが出ないことと正しいことは違う**(レッスン01と同じ)。
   JavaScriptは間違いを黙って受け入れる言語なので、
   自分から厳しくする(`'use strict'`、今ならlinter)必要がある。

2. **スコープは狭いほど強い。** グローバル変数は「誰がいつ書き換えるか
   分からない変数」。読む側は全ファイルを疑う必要がある。関数ローカル
   なら、その関数だけ読めばいい。

3. **現代なら:** `var` すら使わず `let`/`const`、ファイルはES modulesで
   そもそもグローバルを作らない。ESLintの `no-undef` がこの27個を
   一瞬で列挙してくれる。いずれこのプロジェクトでも導入する。

## 余談: 検証方法

「動作を変えていない」ことの確認は、ヘッドレスChromeで
1面・2面・ボス戦・暴れモード・追尾弾・隠しコマンド・勝ち負けの
全パスを実際に遊ばせて行った。このとき、touchJiki を空関数に
差し替えて自機を無敵化するというズルをしたのだが、これができたのは
**全部グローバルだから**。ページのコンソールからゲームの内部状態を
自由にいじれる。デバッグには便利で、チートには無防備。
グローバル設計の功罪を身をもって体験できた。
