# レッスン10: 消す勇気 — gitがあるから捨てられる

対応コミット: `89d6713`

## 何が問題だったか

コードベースには「動いていないのに存在するコード」が溜まっていた。

1. **一度も使われなかった変数**: `bossFlg` は宣言されて以来20年間、
   誰にも読まれず書かれずに鎮座していた。

2. **数え始める前に諦めたカウンタ**: `numBossSh` は宣言だけ存在し、
   実際の増減はすべて `// numBossSh--;` とコメントアウトされていた。
   ボス弾を数える構想があって、途中でやめた形跡。

3. **未完成機能のスタブ**: ハイスコア表示。`txtHiScore` 変数と
   `newTxtHiScore` 関数は完全な形で存在するのに、呼び出しだけが
   `// newTxtHiScore('Hi-Score: 0');` と封印されていた。

4. **試行錯誤の化石**: `// jiki.center().show();`
   `// var that = jiki;` `// jiki.hide();` ——書いては消した跡。

なぜ消せなかったのか、当時の気持ちは分かる。
**「いつか使うかも」「消して壊れたら怖い」**。
バージョン管理がなければ、コメントアウトは唯一の「取っておく」手段だった。

だが今は git がある。消したコードは `git log -p` にすべて残っていて、
いつでも掘り出せる。**コメントアウトは劣化版バージョン管理**であり、
本物を使っている今、役割はもうない。

## 死んだコードの家賃

デッドコードは無害ではない。読む人全員から時間を徴収する。

- `numBossSh` を見た読者は「どこで使われてる?」と検索する(徒労)
- `// numTeki--;` を見た読者は「これは消すべき? 復活させるべき?
  消し忘れ? 意図的?」と当時の意図を推理する(答えは出ない)
- リファクタのたびに「これも直すべき対象?」と判断を迫る

**書いたコードは資産だが、動いていないコードは負債。**

## 消さなかったもの・消す以外の処理をしたもの

- **`// hitJikiSh(...jikiSh2...)`(ボスの当たり判定)** は、
  「後方ショットはボスに当たらない」という仕様の記録だった。
  これはコードの形で置いておく情報ではないので、
  普通の日本語コメントに書き換えた。
  死んだコードに見えて、実は生きた知識だったパターン。

- **`hBossSh`** は「宣言したのに height に `wBossSh` を使っている」
  書き間違いの相方だった。弾は正方形なので実害はないが、
  消すのではなく本来の意図どおり `height : hBossSh` に修正した。
  死んでいたのは変数ではなく、使う側の1文字だった。

## 教え

1. **「いつか使うかも」の"いつか"は来ない。** 来たとしても、
   そのとき git 履歴から掘り出すほうが、20年間コードを
   曇らせ続けるより安い。

2. **消す前に「本当に死んでいるか」を検索で確認する。**
   今回も全識別子を grep してから消した。「たぶん使ってない」で
   消すのは勇気ではなく無謀。確認して消すのが勇気。

3. **コメントアウトに知識が埋まっていることがある。**
   機械的に全削除せず、1つずつ「これは何の跡?」を考える。
   仕様の記録なら日本語コメントに昇格、試行錯誤の化石なら削除。
