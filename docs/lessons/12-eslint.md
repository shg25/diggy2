# レッスン12: 見張りを機械に任せる — ESLint導入

対応コミット: `6134128`

## ESLintとは

コードを**実行せずに**読んで、バグの芽・危険な書き方・規約違反を
指摘する静的解析ツール(linter)。「lint」の語源は衣類の糸くず。
コードに付いた糸くずを取る道具、という意味だ。

これまでのレッスンで人力でやったことの多くは、ESLintのルールに
そのまま対応する。

| レッスン | 人力でやったこと | ESLintのルール |
|---------|----------------|---------------|
| 03 | 宣言なし変数27個を grep と目視で列挙 | `no-undef` |
| 08 | `==` 82箇所を1つずつ精査して置換 | `eqeqeq` |
| 10 | 未使用変数 bossFlg・numBossSh を発見 | `no-unused-vars` |

当時これがあれば、レッスン03〜10の何回分かは
「エディタの赤線を直す」だけで済んでいた。

## 導入したもの

- `package.json` — このプロジェクトが依存する道具の台帳。
  `npm run lint` で検査が走る
- `eslint.config.mjs` — 検査の設定。3つの意思決定が書いてある:
  1. `lib/` は検査しない(借り物は原則いじらない、の徹底)
  2. `DGE` はエンジンが公開するグローバルとして「存在するもの」と教える
  3. 推奨ルールセット + `eqeqeq`(レッスン08の再発防止)
- `.gitignore` に `node_modules`(依存パッケージの実体はコミットしない。
  台帳(package.json)があれば `npm install` で誰でも再現できる)

## 初回実行の結果

```
js/sprite.js
  7:5  error  'bg' is assigned a value but never used  no-unused-vars
```

**一発で見つけた。** `bg` は背景スプライトを入れる変数だが、
入れた後、誰も読んでいなかった。レッスン10で人力の
デッドコード掃除をやった際、「作ったものへの参照は取っておくものだ」
という思い込みで見逃したやつだ。機械には思い込みがない。

面白かったのは、`hitJikiSh` の空ブロック
`if(type === 1){ // レーザー }` が**報告されなかった**こと。
`no-empty` ルールはコメントの入ったブロックを意図的なものとして
許す。道具は思ったより賢い。

## 教え

1. **機械にできる検査は機械に任せ、人間は設計に頭を使う。**
   レッスン08で私は82箇所を1つずつ精査した。あの慎重さは
   「初回の移行」には必要だったが、以後の日常は `eqeqeq` が
   0秒で見張ってくれる。人力レビューは一度きり、lintは毎回。

2. **lintの設定は自分たちの決定の記録。** eslint.config.mjs の
   1行1行は「うちはこう書く」という意思決定だ。ルールを追加する
   ときは、なぜそうするかをコミットメッセージに残す。

3. **導入した日に全部直そうとしない。** 今回はエラー1件だけ
   だったが、大きなプロジェクトだと数百件出る。その場合は
   まず記録し(既存分は一旦許容する設定もある)、新規コードから
   締める。完璧主義は導入の敵。

## 余談

`no-var`(varをlet/constに)というルールもあるが、今日は
有効にしなかった。関数の中身はまだ `var` で書かれていて、
一斉に直すのは次のレッスンの主役だからだ。ルールを入れるのは
コードを直すのと同じコミットで——「赤線だらけで放置」が
一番モラルを下げる。
