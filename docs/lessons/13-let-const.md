# レッスン13: var にさよなら — let/const と「変わらない」という情報

対応コミット: `df454d2`

## var の何が問題だったのか

`var` には現代の目で見ると奇妙な仕様が3つある。

1. **関数スコープ**: ブロック `{}` を無視して、関数全体に染み出す。

```js
for (var i = 0; i < 3; i++) { ... }
console.log(i);   // 3 ← ループの外でまだ生きている!
```

2. **巻き上げ(hoisting)**: 宣言前に使ってもエラーにならず
   undefined になる。「書いた順に実行される」という直感を裏切る。

3. **再宣言し放題**: 同じ関数内で `var x` を2回書いてもエラーに
   ならない。うっかり上書きに気づけない。

有名な事故がループとクロージャの組み合わせだ。

```js
for (var i = 0; i < 3; i++) {
	setTimeout(function() { console.log(i); });  // 3, 3, 3 !!
}
// let にすると 0, 1, 2 (ループの周回ごとに別の i が生まれる)
```

RIDGE部はたまたまこの地雷を踏んでいなかったが、
`setTimeout` とループを多用するゲームコードは常に一歩手前にいた。

## let / const への変換 — 機械的にはやらない

ES2015の `let`/`const` はブロックスコープで、巻き上げも再宣言もない。
変換のルールはひとつ:

**再代入しないなら `const`、するなら `let`。迷ったら `const`。**

今回の34箇所の内訳は const 25 : let 9 だった。つまりこのコードの
変数の7割は「一度決めたら変わらない値」で、`var` はその情報を
読み手から隠していたことになる。`const n = ...` と書いてあれば、
読み手は「n はこの後も同じ値」を前提に読み進められる。
**const は読み手への約束であり、let は「この値は変わるぞ」という警告。**

## ついでに見つけた化石: that退避

```js
var that = sprite;
setTimeout(function() { that.remove(); }, BAN_DURATION_MS);
```

`var that = this;` は当時の定番イディオム(コールバック内で `this` が
変わってしまう問題への対処)だが、ここで退避していたのは `this` では
なく**ただの引数** `sprite`。引数は呼び出しごとに独立していて、
クロージャがそのまま捕まえられるので、退避は最初から不要だった。
イディオムの「形」だけ真似て「理由」を理解していなかった証拠。
`sprite.remove()` を直接呼ぶ形に直した。

## 見張りの追加

eslint.config.mjs に2つのルールを足した。

- `no-var` — var を書いたら即エラー
- `prefer-const` — 再代入していない let を見つけたら「const にしろ」

レッスン12の教えのとおり、**ルールの追加はコードの修正と同じ
コミットで**。赤線ゼロの状態でルールを入れるから、以後の赤線は
すべて「新しく持ち込まれた問題」になる。

## 教え

1. **キーワード選びは情報伝達。** const/let の使い分けは
   コンパイラのためではなく読み手のため。「変わらない」と
   宣言されたものが7割を占めるだけで、コードは格段に追いやすくなる。

2. **イディオムは理由ごと覚える。** `var that = this` の「なぜ」を
   知らずに形だけ使うと、不要な場所にも書いてしまう(そして20年後に
   笑われる)。新しい書き方を覚えるときは、必ず「これは何の問題を
   解決しているのか」とセットで。

3. **古い書き方を駆逐したら、その日のうちに門を閉める。**
   no-var がある限り、var は二度と入ってこない。
