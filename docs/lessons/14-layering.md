# レッスン14: 依存は一方通行に — レイヤー分けと依存性逆転

対応コミット: `2c95d4f`

## 何が問題だったか

レッスン11でモジュール化したとき、importの宣言によって
依存関係が「見える」ようになった。見えたものは、3つの循環だった。

- sprite ⇄ step: スプライトが `goLose()` を呼び、進行役がスプライトを作る
- sprite ⇄ set: スプライトがアイテム効果を呼び、入力がスプライトを撃つ
- stage ⇄ step: ステージ進行が `goCome()` を呼び、進行役がステージを開始する

循環の何が悪いか。**どちらが上か分からない**ことだ。
sprite を理解するには step が要り、step を理解するには sprite が要る。
どこから読み始めても一周させられる。変更の影響範囲も一周する。
テストも一周分まとめてしか書けない。

## 直し方1: 誰にも依存しないものを一番下へ

`stepFlg`・`STEP_*` 定数・`isPlay()` 系は、実は誰にも依存しない
純粋な状態機械だった。それが演出やスプライト生成と同じ step.js に
同居していたせいで、「stepFlg を読みたいだけ」の全員が step.js に
依存させられていた。新モジュール flow.js として最下層に抽出。

## 直し方2: 上向きの「呼び出し」を「通知」に変える

抽出しても残る問題があった。被弾したスプライトは `goLose()` を
呼びたい——だが goLose は演出とタイマーの塊で、明らかに上位層のものだ。
下位が上位を import したら、また循環に戻る。

ここで**依存性逆転(Dependency Inversion)**を使う。

```js
// flow.js (最下層) — フックの「置き場」だけ用意する
export const transitions = {
	come : () => {},   // ボス登場
	battle : () => {}, // ボス戦開始
	lose : () => {}    // 自機被弾
};

// sprite.js (下位) — 「被弾した」と知らせるだけ
transitions.lose();

// step.js (上位) — 何をするかは上位が決めて、起動時に登録
transitions.lose = goLose;
```

sprite は「被弾したら何が起きるか」を**知らなくてよくなった**。
GAME OVER の演出を変えても、スプライトのコードは1文字も変わらない。
「下位は事実を報告し、上位が方針を決める」——会社と同じ構造だ。

## 直し方3: 関数は「作用する対象」の階層に置く

アイテム効果(chJikiSh, chVelJiki, rmGroupTeki)は set.js(入力処理)に
あったが、作用する対象は自機とボス、つまりスプライト層だ。
sprite.js に移動したら、sprite → set の依存が消えた。
「この関数はどこに置くべきか」の答えは、
**その関数が触るデータのそば**にある。

## 結果

依存グラフから循環が消え、きれいな階層になった(機械で検証済み)。

```
ridge (エントリ)
  └ set (入力)
      └ step (場面のオーケストレーション)
          └ stage (ステージ進行)
              └ sprite (エンティティ)
                  └ flow / state / const / text (基盤)
```

新機能を足すときの「置き場所」も自明になる。たとえばハイスコアなら:
記録の保存は基盤層に新モジュール、表示は text、更新のタイミングは step。

## 教え

1. **循環依存は「どちらが上か決めていない」というサイン。**
   コードの問題である前に、設計の未決定事項。まず上下を決める。

2. **下位は事実(何が起きた)を、上位は方針(どうする)を。**
   この分離ができると、方針をいくら変えても下位が揺れない。

3. **見える化 → 問題発見 → 修正、の順番。** レッスン11で
   importが「見える」ようになったから、今日この循環を
   潰せた。見えないものは直せない。

## 余談

transitions フックは最小限の依存性逆転で、本格的にやるなら
イベントバス(emit/on)やオブザーバーパターンになる。
今のゲーム規模には3つのフックで十分。パターンは規模に合わせて
選ぶもので、大きい道具を先に持ち込まない(YAGNI)。
