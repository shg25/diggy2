# レッスン15: 検証を資産に — テストコード

対応コミット: `e498497`

## 種明かし

レッスン03から14まで、「挙動を変えていない」ことをどう保証して
いたか。実は毎回、ヘッドレスChromeでゲームを実際に遊ばせる
検証スクリプトを回していた。1面を無敵で完走し、2面のボスを
暴れさせ、カウンタを監視し、3回連続プレイでリークを測る——
**あれは回帰テストそのもの**だった。

問題は、それが一時的な作業場に住んでいたこと。セッションが
終われば消え、次の人(未来の自分を含む)は使えない。
今日それをリポジトリの資産に昇格させる。

## テストの2層

### 単体テスト (test/unit/) — 速い・狭い

レッスン14の配当がここで受け取れる。flow.js(状態機械)は
DGEにもDOMにも依存しない純粋ロジックになったので、
**ブラウザなしで**検証できる。Node.js組み込みの `node:test` を
使うのでテストフレームワークの追加も不要。

```sh
npm test   # 4テスト、28ミリ秒
```

中身は「isPlay/isFight/isFightBoss の真理値表」など。
表をそのままテストに書けるのは、状態機械が独立しているから。
**テストしやすさは、設計の良さの測定器**である。逆に sprite.js の
単体テストは書けない(DGEが要る)——それはsprite.jsがまだ
エンジンと密結合という設計情報だ。

### E2Eテスト (test/e2e/) — 遅い・広い

実ゲームをChromeで遊ばせる。サーバーの起動から後片付けまで
テスト自身がやるので、コマンド一発で走る。

```sh
npm run test:e2e   # 4シナリオ、約2分
```

- 1面: 撃破でスコアが入り、勝利でタイトルへ戻る
- 2面: ボスが暴れモードで画面を周回し、敗北でタイトルへ戻る
- 敵数カウンタが初回プレイから健全(レッスン05の再発防止)
- 3回プレイしてもプールとDOMが増えない(レッスン07の再発防止)

ゲームとの接点は `window.ridge.debug`(レッスン11で作った
意図的な窓口)だけ。テストが内部実装ではなく公開APIを叩く形に
なっているので、中身をリファクタしてもテストは壊れない。

## 教え

1. **リファクタとテストは鶏と卵。** テストがないとリファクタが
   怖い。しかしテストを書くには構造が必要で、構造を作るには
   リファクタが要る。答えは「E2E(外側)から始める」。外側の
   テストは構造がなくても書ける。実際この14レッスンは
   E2E(drive)に守られて進み、構造ができた今日、
   単体テストが書けるようになった。

2. **バグ修正には再発防止のテストを対にする。** カウンタと
   リークのテストは、レッスン05と07で直したバグの「見張り」だ。
   直したバグが再発するのは、単に直っていなかったバグより士気に悪い。

3. **テストは速さで層を分ける。** 単体(ミリ秒)は毎保存ごと、
   E2E(分)はコミット前。全部E2Eにすると遅くて回さなくなり、
   回さないテストは存在しないのと同じ。

## 余談: テストを書いたら設計の歪みが見つかった

E2Eの初回実行が `numTeki: -17` で落ちた。テストがデバッグAPIで
敵を20体強制湧きさせた際、カウンタの `++` を省いたのが直接の原因
(makeTeki1 は生成のみ、カウントは呼び出し側の責務という契約)。

だがテストを直しながら気づく——**この契約、歪んでいないか?**
makeTeki1 が作った敵は、死ぬとき自分で `numTeki--` する。
なのに生まれるときの `numTeki++` は呼び出し側任せ。減算は自分、
加算は他人。この非対称のせいでテスト(=新しい呼び出し側)が
罠を踏んだ。「テストが書きにくい・間違えやすい」は設計の歪みの
検出器でもある。加算を makeTeki1 側に寄せる修正はいずれ。

## 余談2

「テストを先に書けばよかったのでは?」と当時の自分は思うかも
しれない。だが20年前のあのコードに単体テストは書けなかった
(全部グローバルで、副作用の塊で、切り出せる単位がない)。
書けたのは「ゲームを起動して遊ぶ」というE2Eだけ——そして
それは手動で毎日やっていた。友人と遊んでいたあの時間が、
実は一番贅沢なE2Eテストだったのだ。
