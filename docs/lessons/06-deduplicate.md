# レッスン06: 同じコードは一度だけ — ただし「たまたま似ている」は残す

対応コミット: `fd079aa`

## 何が問題だったか

### その1: 3連コピペの自機ショット生成

`defineJikiSh` は、前方ショット・後方ショット・レーザーの3種を
作るために、**ほぼ同じ10行のループが3回**並んでいた。違いは
画像・サイズ・速度・角度の4つの数字だけ。

コピペの何が悪いか。「あとで直すとき、3箇所直す」からだ。
たとえば画面外に出た弾の処理を変えたくなったら、3箇所の
`.on('ping', ...)` を同じに直す必要があり、1箇所忘れても
エラーは出ない。レッスン05で見た「アイテムの numTeki--」は、
まさにコピペが運んだ消し忘れだった。

「違いの4つの数字」を定義テーブルに(レッスン04の型!)、
「同じ10行」をヘルパー関数に分けた。

```js
var JIKI_SH_DEFS = [
	{ image : 'gfx/jiki/s1.gif', width : 8,  height : 8, velocity : 16, angle : 180 },
	{ image : 'gfx/jiki/s1.gif', width : 8,  height : 8, velocity : 16, angle : 0 },
	{ image : 'gfx/jiki/s3.gif', width : 16, height : 8, velocity : 6,  angle : 180 }
];

function defineJikiSh() {
	pushJikiShots(jikiSh1, numJikiSh1, JIKI_SH_DEFS[0]);
	pushJikiShots(jikiSh2, numJikiSh2, JIKI_SH_DEFS[1]);
	pushJikiShots(jikiSh3, numJikiSh3, JIKI_SH_DEFS[2]);
}
```

### その2: 当たり判定の3連呼び出し

雑魚敵のイベントハンドラには、毎回この3行が現れていた。

```js
hitJikiSh(this, 0, numJikiSh1, jikiSh1, 1);
hitJikiSh(this, 0, numJikiSh2, jikiSh2, 1);
hitJikiSh(this, 1, numJikiSh3, jikiSh3, 0.8);
```

「自機の全ショットとの当たり判定」という**1つの意図**が3行に
分かれて書かれている。`hitAllJikiSh(this, 0.8)` にまとめた。
呼ぶ側は意図だけを書き、詳細はヘルパーが知っている。

## まとめなかったもの: ボスの当たり判定

ボスにも似た3行があるが、よく見ると**仕様が違う**。

```js
hitJikiSh(this, 0, numJikiSh1, jikiSh1, 1);
// hitJikiSh(this, 0, numJikiSh2, jikiSh2, 1);   ← 後方ショットは当たらない
hitJikiSh(this, 1, numJikiSh3, jikiSh3, 0.2);   // レーザーは効きにくい(0.8でなく0.2)
```

後方ショットがボスに当たらないのも、レーザーが効きにくいのも、
おそらく意図したゲームバランス。これを無理に `hitAllJikiSh` に
押し込むと、引数やフラグが増えてヘルパーが濁る。

**重複には2種類ある。**「同じ意図の重複」はまとめる。
「たまたま似ているだけの別物」はまとめない。見分け方は
「片方だけ変えたくなる日が来るか?」。来るなら別物だ。
DRY(Don't Repeat Yourself)は字面の一致ではなく、
**知識・意図の重複**を消す原則。

## 教え

1. **コピペした瞬間は動く。壊れるのは半年後。**
   直すとき・仕様変更のとき、N箇所のうち1箇所を忘れる。
   コピペしたくなったら「違いはどこ?」を考え、違いをデータに、
   共通部分を関数にする。

2. **関数の名前は意図を書くチャンス。**
   `hitAllJikiSh(this, 0.8)` は読み手に「全ショットとの判定」と
   伝える。3行の羅列はそれを毎回読解させる。

3. **まとめすぎは、コピペと同じくらい高くつく。**
   仕様が違うものを1つの関数に押し込むと、if と引数が増えて
   誰も読めなくなる。「共通化しない」も設計判断。

## 余談

検証プレイで、もうひとつ当時からの問題を見つけた。
`defineJikiSh` はゲーム開始のたびに呼ばれ、ショット配列に
**毎回9個の新しいスプライトを追加していく**(古い9個は残ったまま)。
発射処理は配列の先頭3個しか見ないので動きはするが、遊ぶたびに
使われないDOM要素が9個ずつ増える。いわゆるメモリリーク。
挙動保存のため今回のヘルパーも push 方式を忠実に再現している。
「スプライトの寿命管理」はいずれ大きなレッスンになる予定。
